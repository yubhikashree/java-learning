Lesson 8 of Phase 0

OOP — Classes and Objects

Object Oriented Programming is the most important paradigm in software development. Every major application — from Instagram to Google Maps to banking systems — is built with OOP. This is where Java becomes truly powerful.

Part 1 — Class vs Object

The blueprint and the building

Class = Blueprint

Defines what a Student looks like — what data it holds and what it can do. Only exists as a definition. No memory used yet.

Object = Actual instance

A real Student created from the blueprint. Takes up memory. Has its own specific values. You can create hundreds of objects from one class.

Part 2 — Writing your first class

Two files — one class, one main

⚠️ From now on, your programs will have TWO files: Student.java defines the class. Main.java creates objects and uses them. One public class per file — the rule you already know.
// File 1: Student.java public class Student { // Fields — data each student holds String name; int rollNumber; double marks; // Method — behaviour the student has public String getGrade() { if (marks >= 90) return "A"; else if (marks >= 75) return "B"; else if (marks >= 60) return "C"; else return "F"; } public void printReport() { System.out.println("--- Student Report ---"); System.out.println("Name : " + name); System.out.println("Roll No: " + rollNumber); System.out.println("Marks : " + marks); System.out.println("Grade : " + getGrade()); } }
// File 2: Main.java public class Main { public static void main(String[] args) { // Creating objects using 'new' Student s1 = new Student(); s1.name = "Yubhi"; s1.rollNumber = 101; s1.marks = 92.5; Student s2 = new Student(); s2.name = "Arun"; s2.rollNumber = 102; s2.marks = 78.0; // Calling methods on objects s1.printReport(); s2.printReport(); } }
1
Student s1 = new Student() — new creates an object in memory. s1 is a reference variable pointing to that object.
2
s1.name = "Yubhi" — the dot operator accesses a field or method of an object. Same dot you've been using on Strings all along.
3
Methods inside the class don't need static — they belong to an object, not the class itself. This is why static was needed before — now you have real objects.
Part 3 — Constructors

Setting values at the moment of creation

Right now you create an object and then set each field separately. A constructor lets you set everything in one line at the moment of creation — like filling a form when you first register a student.

public class Student { String name; int rollNumber; double marks; // Constructor — same name as class, no return type public Student(String name, int rollNumber, double marks) { this.name = name; this.rollNumber = rollNumber; this.marks = marks; } public String getGrade() { if (marks >= 90) return "A"; else if (marks >= 75) return "B"; else if (marks >= 60) return "C"; else return "F"; } public void printReport() { System.out.println("Name: " + name + " | Roll: " + rollNumber + " | Grade: " + getGrade()); } }
// Main.java — cleaner object creation public class Main { public static void main(String[] args) { // One line per student — much cleaner Student s1 = new Student("Yubhi", 101, 92.5); Student s2 = new Student("Arun", 102, 78.0); Student s3 = new Student("Priya", 103, 85.5); s1.printReport(); s2.printReport(); s3.printReport(); } }
1
this.name = name — this refers to the current object. The left side this.name is the field. The right side name is the parameter. This distinguishes them when they share the same name.
2
Constructor has no return type — not even void. It has the exact same name as the class. Java calls it automatically when you write new Student(...).
Part 4 — Encapsulation

Protecting your data with private + getters/setters

Right now anyone can directly change a student's marks: s1.marks = -50. That's dangerous. Encapsulation means hiding data and controlling access through methods.

public class Student { // private — no direct access from outside private String name; private int rollNumber; private double marks; public Student(String name, int rollNumber, double marks) { this.name = name; this.rollNumber = rollNumber; setMarks(marks); // use setter for validation } // Getter — controlled read access public String getName() { return name; } public int getRollNumber() { return rollNumber; } public double getMarks() { return marks; } // Setter — controlled write access with validation public void setMarks(double marks) { if (marks >= 0 && marks <= 100) { this.marks = marks; } else { System.out.println("Invalid marks. Must be 0–100."); } } }
1
private fields cannot be accessed with s1.marks from outside the class — Java gives a compile error. The only way in is through the getter/setter methods you provide.
2
The setter validates input before storing it. s1.setMarks(-50) prints an error instead of corrupting the data. This is the real power of encapsulation.
💡 Industry standard: Always make fields private. Always provide public getters and setters. This is the first thing senior developers check in a code review.
Part 5 — Array of objects

Managing multiple students efficiently

public class Main { public static void main(String[] args) { // Array of Student objects Student[] students = new Student[3]; students[0] = new Student("Yubhi", 101, 92.5); students[1] = new Student("Arun", 102, 78.0); students[2] = new Student("Priya", 103, 85.5); // Loop through and print all reports for (Student s : students) { s.printReport(); } // Find topper Student topper = students[0]; for (int i = 1; i < students.length; i++) { if (students[i].getMarks() > topper.getMarks()) { topper = students[i]; } } System.out.println("Topper: " + topper.getName()); } }
1
An array of objects works exactly like an array of ints — but each slot holds an object reference instead of a number.
2
The topper-finding logic is the same index-tracking pattern from Lesson 6 — but now comparing objects using their getter methods.
